# 断点状态与恢复规则

## 目录

1. 文件
2. 状态内容
3. 写入时机
4. 恢复算法
5. 安全约束

## 1. 文件

每本书目录保存：

- `生产状态.json`：机器可读的唯一状态源
- `生产日志.md`：只追加的人类可读事件日志

使用 `scripts/checkpoint.py` 原子写入 JSON；正常流程不要手工编辑。

## 2. 状态内容

状态包含：

- 配置文件路径、非敏感账号提示、书名、作者、工程包、输出目录、目标字数和章节总数
- GPTs 对话 URL、番茄草稿 URL 与草稿 ID
- 当前步骤、下一动作、已完成步骤
- 用户选题、选题确认状态、发布授权状态
- 每章标题、阶段、纯文本路径、Word 路径和复核状态
- Word 模板、选题原文、冻结策划、复核报告、最终正文、合并 DOCX、番茄上传源和封面路径
- 番茄正文、章节格式、AI 标记、分类、试读比例和草稿保存状态
- 是否提交审核、是否发布及最后更新时间

章节阶段按以下顺序单调推进：

`pending → generating → generated → text_saved → word_saved → reviewed → finalized`

## 3. 写入时机

在以下动作完成后、下一动作开始前写入：

- 工程包已交付 GPTs
- 选题已生成并展示
- 用户已确认选题
- 项目策划已冻结
- 每章生成、文本保存、Word 保存或最终修订
- GPTs 全书复核完成
- 合并 DOCX 完成
- 番茄草稿创建、正文上传、章节格式完成
- 封面完成或上传
- AI 标记、分类、试读比例完成
- 草稿保存完成
- 用户明确授权审核或发布

## 4. 恢复算法

1. 验证断点登记的使用者配置仍存在且有效；账号提示不一致时暂停确认。
2. 运行 `resume` 读取当前步骤和下一动作。
3. 运行 `verify` 检查所有已登记本地文件。
4. 已登记且存在的最终文件不得重复生成。
5. 正式书名冻结后，使用 `relocate` 将临时项目目录迁移到书名目录；不得靠手工移动后猜测路径。
6. 已记录 GPTs URL 时重新打开同一对话，不新建创作对话。
7. 已记录番茄草稿 URL/ID 时恢复该草稿，不新建重复草稿。
8. 若状态声称完成但文件缺失，将流程回退到最早不一致步骤并写日志。
9. 若文件存在但状态落后，核对文件非空后将状态补进，不覆盖文件。
10. 从 `next_action` 继续；不得仅凭聊天记忆猜测。

## 5. 安全约束

- `publish_authorized` 默认 `false`。
- `submitted_for_review` 和 `published` 默认 `false`。
- 未记录用户授权证据时，脚本拒绝标记提交或发布。
- 自动发送 GPTs 提示不等于发布授权。
- 断网、浏览器重启或 Codex 新线程均不得清除出版保护字段。
